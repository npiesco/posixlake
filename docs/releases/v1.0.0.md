# posixlake v1.0.0 Release Notes

## Summary

- Version: `v1.0.0`
- Release date: TBD
- Highlights:
  - enterprise-readiness baseline completed for the self-hosted CLI workflow
  - release gate expanded across formatting, lint, build, unit, integration, audit, and SBOM jobs
  - operational health, metrics, SLO, runbook, and failure-drill coverage added
  - compatibility, upgrade safety, rollback validation, and deprecation policy documented

## Compatibility

- Supported OS/architectures:
  - Linux `x86_64`
  - macOS `x86_64`, `aarch64`
  - Windows `x86_64`, `aarch64`
- Python versions:
  - source installs: Python `3.8` through `3.12`
  - release packaging path: Python `3.11`
  - prebuilt wheel consumption: Python `3.11+` recommended
- Upgrade notes:
  - legacy `0.0.x`-style metadata layouts without newer optional fields such as `primary_key` are supported for open/query and metadata upgrade
  - take a verified backup before any upgrade
- Rollback notes:
  - rollback is supported through documented backup/restore flow
  - in-place downgrade is not supported

## Added

- CLI `health` and `metrics` commands with JSON output
- documented SLOs and operational runbooks
- failure-drill and operations CLI integration coverage
- compatibility matrix and deprecation process docs
- support statement, release policy, and production-readiness checklist
- Sigstore keyless signing and checksum generation for published Python artifacts

## Changed

- release workflow now treats broader integration coverage as part of the required release gate
- published Python artifacts now include checksum and signing outputs in the automated release bundle
- readiness documentation now explicitly defines support boundaries, compatibility expectations, and release criteria

## Deprecated

- No removals in `v1.0.0`
- Deprecation policy now requires at least one minor release of warning before removal for supported features

## Fixed

- auth-enabled open paths now fail closed across previously permissive metadata and admin operations
- upgrade compatibility coverage now validates legacy metadata open, metadata upgrade, and rollback restoration paths
- failure recovery coverage now includes delta-log corruption handling, restore verification, and concurrent write safety drills

## Known Limitations

- full integration coverage remains Linux-only in the required CI gate; macOS and Windows currently receive build and unit-test coverage
- mixed-version concurrent writers are not supported
- in-place downgrade is not supported
- sign-off owners in `PROD_READINESS.md` are still administrative follow-up items
- audit logs are append-only by convention and filesystem permissions, not cryptographically tamper-evident

## Upgrade Checklist

1. Verify backup integrity.
2. Review `docs/compatibility.md` and upgrade/rollback runbook.
3. Apply the upgrade in staging first.
4. Validate `posixlake-cli health`, `posixlake-cli metrics`, and representative query paths.
5. Retain rollback backup until post-upgrade verification is complete.
